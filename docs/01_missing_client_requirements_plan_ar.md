# خطة تنفيذ نواقص طلبات العميل

تاريخ الإنشاء: 2026-06-06

## الهدف

استكمال الفجوات التي ظهرت بعد مراجعة فيديو العميل ومقارنتها بمشروع `driver` الحالي، مع الحفاظ على معمارية المشروع: `Views` للعرض، `Controllers` للحالة والتفاعل، `Repositories` لعمليات Firebase، و`Models` للبيانات.

## ملخص الحالة الحالية

المشروع يحتوي بالفعل على شاشات: الرئيسية، تفاصيل الطلب، المحفظة، الدعم الفني، الإشعارات، والحساب الشخصي. النواقص المتبقية ليست بناء شاشات من الصفر، بل إضافة سلوكيات تشغيلية مهمة وربطها بـ Firebase/Storage.

## أولويات التنفيذ

### 1. الصفحة الرئيسية والطلبات

- تغيير تبويب الطلبات الأوسط من `النشطة` إلى `الحالية`.
- تشغيل تنبيه صوتي عند وصول طلب جديد:
  - داخل التطبيق: تشغيل صوت متكرر عند ظهور طلب جديد في stream الطلبات.
  - خارج التطبيق: تجهيز قناة إشعار محلية بصوت قوي، مع ملاحظة أن الرنين المستمر في الخلفية يحتاج صلاحيات/Native handling حسب قيود Android و iOS.

معايير القبول:

- التبويب يظهر باسم `الحالية`.
- وصول طلب جديد أثناء فتح التطبيق يشغل منبهًا يمكن إيقافه عند قبول/إخفاء الطلب أو فتح الإشعار.
- إشعار الطلب الجديد يستخدم قناة عالية الأهمية.

### 2. تفاصيل الطلب

- استبدال فتح الملاحظة الصوتية كرابط خارجي بمشغل صوت مدمج داخل البطاقة.
- إضافة إثبات الاستلام بالصورة:
  - عند `تأكيد استلام الطلب` تفتح الكاميرا.
  - ترفع الصورة إلى Firebase Storage.
  - تحفظ رابط الصورة داخل وثيقة الطلب.
  - تسجل الرابط في `order_events.metadata`.
- إضافة إثبات التسليم بالصورة بنفس الآلية عند `إنهاء الطلب`.
- إضافة زر دعم سريع داخل تفاصيل الطلب مربوط برقم الطلب.

معايير القبول:

- الملاحظة الصوتية تعمل Play/Pause مع progress bar.
- لا يتم تحديث حالة الاستلام أو التسليم بدون صورة إثبات.
- صور الإثبات محفوظة في Storage ومرتبطة بالطلب.

### 3. عمولة التطبيق والمحفظة

- خصم عمولة التطبيق عند قبول الطلب داخل نفس Firestore transaction الخاصة بالقبول.
- منع قبول الطلب إذا كان رصيد السائق أقل من العمولة.
- إنشاء wallet transaction حتمية ID لمنع خصم العمولة مرتين.
- إضافة طلب شحن المحفظة:
  - زر `شحن`.
  - إدخال رقم العملية/التحويل.
  - التقاط/اختيار صورة إيصال التحويل.
  - رفع الإيصال إلى Firebase Storage.
  - إنشاء طلب مراجعة في collection مخصص للإدارة.

معايير القبول:

- قبول الطلب، خصم العمولة، وإنشاء سجل المحفظة يحدثون معًا أو لا يحدث أي منهم.
- لا يمكن تكرار خصم العمولة لنفس الطلب والسائق.
- طلب الشحن يظهر للإدارة كـ `pending` مع رقم العملية ورابط الإيصال.

### 4. الدعم الفني

- تحديث أسباب الدعم إلى:
  - استفسار عام
  - شكاوى
  - مشاكل تقنية
  - حالات طارئة
  - مشاكل في الدفع
  - شحن المحفظة
  - العميل لم يدفع
  - أخرى
- دعم فتح الشاشة من تفاصيل الطلب مع تمرير `orderId`.
- حفظ `order_id` داخل تذكرة الدعم عند وجوده.

معايير القبول:

- تذكرة الدعم العامة تعمل كما هي.
- تذكرة الدعم القادمة من طلب تحمل رقم الطلب.

### 5. الحساب الشخصي

- إعادة تنسيق إحصائيات القبول والإلغاء كنسب مئوية واضحة وملونة.
- تحسين عرض بيانات المركبة والرخصة إن كانت موجودة في بيانات السائق.

معايير القبول:

- النسب لا تكسر الواجهة إذا لم تكن البيانات موجودة.
- بيانات المركبة والرخصة تظهر فقط عند توفرها.

## ملاحظات Backend و Firebase Rules

- يجب السماح للسائق بكتابة صور الإثبات داخل طلبه فقط.
- يجب السماح برفع الصور إلى مسارات:
  - `order_proofs/{orderId}/...`
  - `wallet_recharge_requests/{driverId}/...`
- يجب أن تراجع الإدارة طلبات شحن المحفظة وتحوّلها لاحقًا إلى wallet topup مع transaction مستقل.
- الرنين المستمر خارج التطبيق قد يحتاج Android foreground service أو full-screen intent إذا أراده العميل بنفس سلوك تطبيقات التوصيل الثقيلة.
